AI & Software Engineering

three observations

01

The Job Changed

Pre-AI
programming + architecture
intertwined — programming and architecture happen together, in flow
Post-AI
build
review & refine
swearing at robot
architecture
fragmented — same work, different composition, less flow

The flow state was a feature, not a bug.

02

What's Scarce Now

There was always bad code.
The difference is you used to be able to tell.

Writing code
easy
Knowing what
to build
hard
Telling good
from bad
hard

What's hard now

Experience helps — but let's be honest,
even experienced devs struggle to tell if AI code is solid.

03

Java Collections → AI

Before 1998

Hand-rolled data structures.

Plenty of bad code. But you could tell it was bad.

JDK 1.2

ArrayList, HashMap, TreeSet.

10x more accessible. 10x more dangerous.

Clean, correct code
10x faster
←
Collections
→
Broken code that
looks fine

list.contains(x) in a loop. LinkedList.get(i) in a hot path.

Looked fine. Tests passed. Broke under load.

03

Same Pattern. Different Failure.

Pre-AI garbage

Looked like garbage.

You could spot it, measure it, fix it.

AI garbage

Looks like good code.

Reads well. Runs fine. Almost impossible to tell apart.

There was always garbage.
Now it's wearing a suit.

04

All Roads Lead to Rome

They all arrive. Not all of them should have been built.

It looks right. It runs. It passes tests.
That used to be enough.

05

What Changes Next

The skill becomes: telling good from bad, fast.

Less process upfront. More refinement after.
We don't fully know what that looks like yet.

Nobody has this figured out yet.
